## 1. Executive summary

RestoFlow is in **far better shape than its own documentation says**. The core loop — owner onboarding → device pairing → menu → POS order → pay-now/pay-later → receipt print (Wi-Fi + in-house Bluetooth SPP, Arabic/Hebrew raster) → KDS ticket → statuses → shift close → owner reports — is real, fail-closed, trilingual (ar/he/en, Arabic default), and heavily tested (**2,266 Dart tests + 187 pgTAP files, all green; analyzer clean; all four guards pass**). Backend security is unusually disciplined: every SECURITY DEFINER function pins `search_path`, every public RPC is revoke-from-public/grant-authenticated-only, tenant scoping is consistent and GUC-free, money is integer-minor and server-authoritative, and the KDS money-free invariant is structurally enforced end to end.

**Verdict: READY WITH WARNINGS** for a *supervised, cash-only, tax-off, single-branch, 1–3 day* pilot — and **not ready unsupervised**. The genuine gates are few and small:

1. **Deployment state, not code:** 4 local migrations (customer name, kitchen prep/meat counts, dashboard order history) are not applied to hosted; APKs built from `main` already call them. Without the R-003-gated hosted apply, the Dashboard **Orders tab errors on day one** and KDS kitchen counts / customer names silently vanish.
2. **Three small verified client defects** that would visibly hurt a real service day: a double-tap on Send creates **duplicate kitchen orders** (no in-flight guard); KDS columns sort by **random UUID instead of arrival time** (FIFO broken); a POS/KDS launched without internet is **bricked behind a retry-less developer help page**.
3. **Three cash-trust gaps:** opening float hard-coded to 0 (every close shows a fake variance), no reachable **void** for a wrongly-entered order, and a payment retry after a timeout mints a **new idempotency key** so a genuinely-paid order can show "unpaid" forever.
4. **Process gaps:** CI silently skips ~47 test files including the entire printing stack (all pass locally — verified in this audit); the real-mode APK build recipe is un-scripted; there are **zero Arabic user guides** for the staff who will run the pilot; and the repo's own readiness docs are dangerously stale (they still claim "POS is a scaffold").

Everything above is small (mostly client-only, one deployment task, one docs task). With the P0 batch in §19 done — roughly **2–4 focused days** — the pilot is a reasonable bet.

## 4. Pilot readiness verdict

**READY WITH WARNINGS** for a supervised, cash-only, tax-off, ILS-only, single-branch, 1 POS + 1 KDS, 1–3 day pilot **with the owner/developer on site** — conditional on completing the P0 items in §5/§19 first.

**NOT ready** for: unsupervised operation, unstable-internet venues (payments/PIN/menu are online-only; only order submission is durably queued), tax-enabled operation (three verified tax inconsistencies — keep tax OFF), multi-station kitchens (no per-item station routing), or anything requiring refunds/corrections of paid orders (deliberately deferred, D-023/D-024).

Why "ready": the happy path is complete and honest end to end; failures fail closed and visibly (no fabricated data anywhere — verified repeatedly across every seam); money cannot be corrupted by any found defect (server guards hold in every failure scenario found); security posture (RLS, device auth, PIN lockout, anon-key-only clients) is appropriate.

Why "warnings": the found defects are exactly the kind a real service day surfaces — double-taps, wifi blips, printer-off moments, day-2 board state, cash-drawer trust — and there is currently **no correction path for mistakes** (no void) and **no user documentation** for the Arabic-speaking staff.

## 5. Top blockers before a real restaurant pilot

Ordered by risk-to-pilot. B1 is a deployment task; B2–B5 are small verified code fixes; B6–B8 are process.

1. **B1 — Hosted DB ↔ APK contract drift (verified locally; hosted state must be confirmed by the human).** Hosted was last applied through `rf_report_004` on 2026-07-06 (recorded in [DEPLOYMENT.md §13](../DEPLOYMENT.md)). Four newer local migrations — `20260707120000_order_customer_001`, `20260708090000_kitchen_prep_001`, `20260709090000_kitchen_meat_001`, `20260710090000_orders_history_001` — each carry an explicit "do NOT apply to hosted (R-003 gate)" header and have **no apply record anywhere in docs/**. Clients at HEAD already call them: the Dashboard Orders tab has **no missing-RPC fallback** ([real_order_history_repository.dart:64-71](../../apps/dashboard/lib/src/data/real_order_history_repository.dart)) and will render a permanent error; customer names and kitchen prep/meat counts silently vanish. Required with it: confirm **anonymous sign-ins are ON** on hosted (POS/KDS device auth hard-depends on it) and record the human R-003 sign-off. Verify with `supabase migration list --linked` first — network access was prohibited in this audit.
2. **B2 — POS double-submit race (verified).** `canSend` has no in-flight flag ([cart_panel.dart:104](../../apps/pos/lib/src/widgets/cart_panel.dart)) while real-mode submit awaits the full network push ([outbox_controller.dart:289-296](../../apps/pos/lib/src/state/outbox_controller.dart)); no spinner appears. A double-tap on slow Wi-Fi mints fresh order/op UUIDs, so D-022 idempotency cannot dedupe — **two orders reach the kitchen**, and (see B5) there is no void to clean up. Cashiers reliably double-tap unresponsive buttons.
3. **B3 — KDS FIFO broken (verified).** Columns sort by `kitchenTicketId` = `'$orderId:$station'` where orderId is a random v4 UUID ([kds_board.dart:71](../../apps/kds/lib/src/widgets/kds_board.dart), [kds_ticket_mapper.dart:233](../../packages/feature_kitchen/lib/src/kds_ticket_mapper.dart)) — order within New/Preparing is random vs arrival; kitchens work top-down. `submittedAt` is already on the view; the fix is a tiny client sort.
4. **B4 — Offline-launch brick (verified).** Every real-mode start does a live `signInAnonymously()`; any failure (including plain no-wifi) renders a static developer help page ("anonymous sign-in is disabled… supabase/config.toml…") with **no retry action** ([apps/pos/lib/main.dart:217-221](../../apps/pos/lib/main.dart), same in KDS; [device_sign_in_unavailable_view.dart](../../packages/feature_auth/lib/src/device_sign_in_unavailable_view.dart)). A tablet reboot on flaky wifi = force-close-and-relaunch until wifi returns. Related trap: a transient failure during session restore silently drops a paired device to the pairing screen; typing a new code re-pairs it as a new identity and orphans the old outbox queue.
5. **B5 — No void for a wrongly-entered order (verified).** `app.void_order` exists server-side (RF-053/062, audited, blocks voiding paid orders) but is **unreachable**: the newest `sync_push` op allowlist has no `order.void` and no UI calls it. A mis-keyed order — a daily event — cooks in the kitchen and permanently inflates billed totals with zero recourse.
6. **B6 — CI test gaps + unscripted APK builds (verified).** `.github/workflows/ci.yml` hand-enumerates suites and skips ~47 test files across `packages/printing` (17, incl. ESC/POS goldens + RTL raster), `feature_menu` (18), `sync`, `data_remote`, `native_printing`, `tools/print_bridge` — printing is the highest-churn pilot-critical area. (Good news: **this audit ran all the skipped Flutter suites locally — all pass.**) Separately, no committed script builds the real-mode APKs; the `supabasekey.env.local` → `RESTOFLOW_*` mapping exists only in session memory, one forgotten dart-define silently ships a **demo** APK (POS/KDS never consume the existing `isDemoModeMisconfigured` guard), and release APKs are debug-key signed on one machine (a rebuild elsewhere forces uninstall → wipes pairing + any unsent offline orders).
7. **B7 — On-hardware print validation has never happened.** The BT SPP path is days old and has never been proven on the pilot's actual printer model. Also verified: the whole live path hardcodes **80mm/48col/576 dots** (`PrinterProfile.escPos58mm` exists but nothing can select it) — if the pilot printer is 58mm, receipts are clipped/garbled with no in-app remedy; and the setup **test print never exercises the raster path** that every real Arabic receipt uses (it's ASCII-only, and the BT test page even prints "Transport: network (TCP)").
8. **B8 — Zero user-facing documentation, and the repo's readiness docs are wrong.** No cashier/kitchen/owner guide exists in any language (pilot staff are Arabic-speaking). Meanwhile `PILOT_READINESS.md` (2026-06-23) answers its own question with "REAL PILOT: BLOCKED — apps/pos is a scaffold", `ANDROID_BUILD.md` §7b still recommends re-adding the exact Bluetooth plugin family that was removed for corrupting raster prints, and `DEPLOYMENT.md` §13's preflight ("exactly ONE pending migration expected… STOP") is now factually wrong about the pending set.

## 6. Top non-blocking risks (pilot will feel these; won't corrupt anything)

1. **Fake cash variance every close (verified, high):** shift auto-opens with `const openingFloatMinor = 0` ([pos_session.dart:267](../../apps/pos/lib/src/state/pos_session.dart)); a real drawer float of ₪300 reports +₪300 variance daily and RF-055 forces a reason to be typed. Server already accepts `opening_float_minor` — client-only fix.
2. **Paid-but-shows-unpaid after a timeout retry (verified, high):** `recordCashPayment` mints a new `local_operation_id` per call ([payment_repository.dart:235](../../apps/pos/lib/src/data/payment_repository.dart)); if the first attempt applied but the response was lost, the retry is rejected ("already has a completed payment") and the POS shows unpaid forever, with copy asserting "the order stays unpaid" — false. No double-charge is possible (server guard holds); this is a truth/trust defect.
3. **Outage recovery is manual (verified, high):** transport failures are marked `rejected` like real rejections and auto-retry stops after 5 attempts (~2 min); after wifi returns someone must tap "retry all". Payments/PIN/menu/shift ops have no queue at all — mid-shift wifi loss stops receipts and payment recording (pay-later is the workaround).
4. **Missed-order signals in the kitchen:** no sound on new-ticket arrival, and the KDS never keeps the screen awake (zero wakelock anywhere) — a tablet with a 30s display timeout goes dark mid-service. Both one-liners-ish.
5. **Day-2 board pollution (verified, medium):** no time window anywhere in the pull/mapper pipeline, and cursors are in-memory — yesterday's un-bumped orders reappear every morning as 1440-minute "New" tickets (elapsed pill shows raw minutes).
6. **8h PIN session expiry mid-day (verified, medium):** a 9am sign-in dies ~5pm; KDS handles it (re-auth screen), POS just fails pushes generically with no "sign in again" routing.
7. **Live print-failure copy is misleading (verified, medium):** a printer that's simply off shows "Print bridge unavailable" (there is no bridge on the native path) and every other category shows generic "Print failed" — while correct category-specific ar/en/he copy already exists, used only by the settings test print.
8. **Real receipts have no restaurant identity (verified, medium):** header is the generic word "Receipt"/"إيصال"; the owner will notice on day one. Tenant/branch name is already on the device.
9. **KDS offline status advances silently revert (verified, medium):** fire-and-forget pushes are swallowed; after a blip, bumped tickets snap back with no message (print-on-acknowledge correctly doesn't fire).
10. **Misc verified paper-cuts:** BT settings Test/Save silently no-op for a previously-saved printer until the row is re-tapped; editing a cart line while the menu is unavailable strips its paid modifiers (silent under-charge, narrow window); recent-orders shows stale pre-discount totals for pay-later; `kRestoflowInk3` fails WCAG small-text contrast (~2.8:1); "1 عناصر" (no ICU plurals) on every single-item order; browser-print HTML lacks `dir="rtl"`; a possible native write-phase wedge (no write watchdog — a stalled printer can block the single-thread job executor until app restart; Dart side stays honest via the outer timeout).

## 10. Printing audit (Domain E)

## 12. Money / payment / tax audit (Domain G)

## 13. Offline / sync / outbox audit (Domain H)

**Verdict: offline-TOLERANT, not offline-FIRST.** The one genuinely offline-capable path is POS **order submission** mid-service with a live in-memory PIN session: durable shared_preferences outbox keyed `(deviceId, localOperationId)`, survives refresh/restart, replays idempotently against a solid server ledger. Everything else needs the network: **boot** (anonymous sign-in → pairing restore → PIN → menu are all live calls with no cache — B4), **payments**, **shift ops**, and the **KDS** (pull-only 5s polling; status pushes fire-and-forget). **Verified failure scenarios to brief:** POS/KDS restart during an outage = unusable until wifi returns (B4); wifi loss mid-shift = orders queue, payments/receipts/PIN wait (pay-later + retry is the answer); outage >~2 min strands queued orders as "failed" until a manual retry-all (§6.3 — the transport already classifies transient vs server, the policy just ignores it); kitchen bumps during a blip silently revert; the 8h PIN expiry lands inside a normal service day with no POS re-auth routing (§6.6); recent-orders paid/unpaid state is device-local truth that can drift from server truth with no reconciliation pull (the timeout-retry race in §6.2 is the trigger); durable outbox never GCs applied entries (harmless for days, unbounded over weeks); KDS re-pulls the branch's entire history every launch (fine at pilot scale, deliberate RF-063 call); device sessions hard-expire at 7 days with no renewal — **all devices paired on setup day bounce to the pairing screen together on day 7** (plan a re-pair if the pilot extends).
