07 · Principles

المبادئ المعمارية

قرارات ثابتة تحكم كل مرحلة — أمان وعزل وموثوقية مبنية في التصميم، لا مضافة لاحقًا.

Multi-Tenant Isolation

كل صف يحمل tenant_id / restaurant_id؛ مطعم لا يقرأ بيانات مطعم آخر — حتى لو عُدّل الطلب يدويًا، القاعدة ترفض.

Per-Person Identity

لا حساب مشترك: كل شخص حساب، كل جهاز هوية، كل عملية باسم صاحبها الحقيقي مع إلغاء وصول فوري.

Four Security Layers

RLS · Membership & Roles · Server-side functions (fn_apply_discount) · Database constraints صارمة.

Offline-First

SQLite محلي + Outbox + Idempotency: المطعم لا يتوقف، والمزامنة تتم لحالها عند رجوع النت.

Bug Prevention by Design

State machines · Audit log لا يُعبث به · Typed contracts بين Flutter والـ Backend.

Isolation Tests

اختبارات إلزامية: مطعم لا يقرأ بيانات مطعم آخر، وموظف مُلغى لا يشتغل — شرط لكل دمج.

10 · Risks

المخاطر والأسئلة المفتوحة

R1
تكامل الأجهزة (ESC/POS)
اختلاف الطابعات والموديلات. التخفيف: تثبيت موديل واحد في M3 قبل التوسّع.
R2
تعارضات المزامنة Offline
رجوع النت بطلبات متراكمة. التخفيف: Idempotency + Outbox مُختبَران في M2.
R3
صحّة الـ RLS والعزل
خطأ بسيط = تسريب بيانات بين مطاعم. التخفيف: اختبارات عزل إلزامية + sign-off بشري.
R4
تمدّد النطاق (Scope creep)
انجرار نحو المؤجّل. التخفيف: قائمة «مؤجّل» مُجمّدة + قرار بشري على الـ scope.
R5
الاعتماد على منفّذ واحد (bus factor)
بشري واحد + وكلاء. التخفيف: توثيق كل قرار + Codex كمراجع مستقل + Git كمصدر حقيقة.
