Plan Overview

This document defines who does what across a mixed team of one human and three AI agents, with clear delivery milestones and a proposed schedule from M0 to M4.

07 · Principles

Architecture Principles

Fixed decisions governing every phase — security, isolation and reliability built into the design, not bolted on later.

Multi-Tenant Isolation

Every row carries tenant_id / restaurant_id; one restaurant can't read another's data — even if the request is hand-edited, the database refuses.

Per-Person Identity

No shared account: each person an account, each device an identity, every action under its real owner with instant access revocation.

Four Security Layers

RLS · Membership & Roles · server-side functions (fn_apply_discount) · strict database constraints.

Offline-First

Local SQLite + Outbox + idempotency: the restaurant never stops, and sync happens on its own when the network returns.

Bug Prevention by Design

State machines · tamper-proof audit log · typed contracts between Flutter and the backend.

Isolation Tests

Mandatory tests: a restaurant can't read another's data, and a revoked employee can't operate — a gate on every merge.

10 · Risks

Risks & Open Questions

R1
Hardware integration (ESC/POS)
Printer & model variation. Mitigation: standardize one model in M3 before scaling.
R2
Offline sync conflicts
Network returns with queued orders. Mitigation: idempotency + Outbox tested in M2.
R3
RLS correctness & isolation
A small bug = data leak between restaurants. Mitigation: mandatory isolation tests + human sign-off.
R4
Scope creep
Drifting into deferred items. Mitigation: frozen "deferred" list + human scope decision.
R5
Single-builder bus factor
One human + agents. Mitigation: document every decision + Codex as independent reviewer + Git as source of truth.
